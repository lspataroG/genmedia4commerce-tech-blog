## The Use Case

Product 360-degree spinning videos are one of the highest-impact media types in e-commerce. They let shoppers view a product from every angle, dramatically increasing purchase confidence and reducing returns.

Traditionally, creating a 360 spin requires a motorized turntable, controlled lighting, and a camera capturing dozens of frames per rotation — followed by post-production stitching. This costs $100–500+ per product and limits coverage to high-value SKUs.

The GenMedia approach: take 1–4 static product photographs and generate a smooth, continuous 360-degree spinning video using **Veo 3.1** (Google's video generation model) with reference-image conditioning. No turntable, no studio, no post-production. The pipeline produces an 8-second, 16:9 video of the product rotating clockwise on a white background.

This section walks through the generic product spinning pipeline — the version that handles all non-footwear products (mugs, backpacks, electronics, watches, etc.). The specialized footwear pipeline is covered in **[Part 6](genmedia_at_scale_spinning_shoes.md)**.

---

## The Reference-to-Video (R2V) Approach

The pipeline uses Veo 3.1's **reference image conditioning** capability. Instead of describing the product in text and hoping the model imagines it correctly, we provide actual product photographs as visual references. The model generates a video that features the product from those references.

This aligns with the architecture's core principle: **let reference images lead, keep descriptions minimal.** The model receives:

- 1–3 reference image canvases showing the product from different angles
- A short text prompt describing the action ("a continuous 360-degree orbit") and scene ("a completely white studio void")
- A brief subject description ("a red ceramic mug") generated by Gemini from the reference images

The text prompt never describes product details. It describes the *scene and motion*. The product details come from the images.

### Step 1: Input Preprocessing

The pipeline accepts 1–4 product images and processes each through the extraction and enhancement toolkit described in [Part 3](genmedia_at_scale_input_processing.md).

**Per-image processing (in parallel):**

1. **Background removal:** Vertex AI Image Segmentation isolates the product, with rembg as fallback. The product is composited onto a white background.
2. **Upscaling:** Imagen 4.0 upscales by up to 4x (auto-downgrading for large images).
3. **Artifact cleanup:** The original segmentation mask is reapplied after upscaling to remove edge artifacts introduced by the upscaler.

**Canvas creation:**

After individual preprocessing, images are placed on standardized 4K canvases (3840x2160):

| Input Count | Canvas Strategy |
|-------------|-----------------|
| 1–3 images | One canvas per image |
| 4 images | Images 1 and 2 get individual canvases; images 3 and 4 are stacked side-by-side on a shared canvas |

This produces a maximum of 3 reference canvases — the practical limit for Veo reference image conditioning. Each canvas has consistent margins (216px top/bottom, 384px left/right) and all products are scaled to the same height.

### Step 2: Description Generation

A vision model (Gemini 2.5 Flash) analyzes the original input images and generates a short product description.

**Configuration:**
- Temperature: 0 (deterministic — same images always produce the same description)
- Max output tokens: 100
- No chain-of-thought reasoning

**Output format:** `"A [color] [product type] standing still in a completely white studio void"`

The description is deliberately minimal. It identifies the product type and primary color — nothing more. No brand names (even if visible), no structural details, no feature descriptions. This follows the architecture principle: let the images carry the visual information, use text only to anchor the scene.

The description is inserted into a structured prompt template with three sections: Subject, Action, and Scene.

### Step 3: Video Generation

The preprocessed canvases and prompt are sent to **Veo 3.1** for video generation.

**Configuration:**

| Parameter | Value |
|-----------|-------|
| Model | `veo-3.1-generate-preview` |
| Duration | 8 seconds |
| Aspect ratio | 16:9 |
| Audio | Disabled |
| Reference type | `asset` (visual reference) |

The generation call is asynchronous — it returns an operation ID that is polled every 2 seconds until the video is ready. The result is raw MP4 bytes.

### Step 4: Two-Level Validation

Every generated video passes through two sequential validation checks. If either fails, the video is regenerated. The pipeline allows up to **4 total attempts** (1 initial + 3 retries).

The checks are deliberately **ordered by cost**: the free optical flow check runs first, and the paid Gemini evaluation only runs if optical flow passes.

```
For each attempt (max 4):
  1. Generate video
  2. Rotation direction check (optical flow) → if not clockwise, retry
  3. Glitch detection (Gemini) → if glitches found, retry
  4. If both pass → accept
```

If all 4 attempts fail validation, the last generated video is used as a best-effort result. The response includes metadata indicating the validation status.

## Validation: Level 1 — Rotation Direction

The first validation check uses **sparse optical flow** (Lucas-Kanade) to determine whether the video shows clockwise rotation. This is a pure computer vision approach with no API calls — it runs locally on the video frames.

### How It Works

1. **Feature detection:** Trackable points are identified in each frame (up to 100 corners per frame)
2. **Flow tracking:** Features are tracked across consecutive frames, computing horizontal displacement vectors
3. **Motion aggregation:** Average horizontal displacement per frame is computed — negative values indicate clockwise motion, positive values indicate anticlockwise
4. **Segment analysis:** Consecutive frames are grouped into directional segments using sliding windows (24-frame windows, 50% overlap)
5. **Classification:** A rule-based classifier evaluates the segment pattern:

| Rule | What It Detects | Result |
|------|-----------------|--------|
| Motion spike | Any displacement > 32px horizontal or 35px vertical | Invalid (scene jump) |
| Direction change | Significant minority direction with high strength | Invalid (reversal mid-spin) |
| High ratio | Minority direction ratio > 0.45 | Invalid (ambiguous rotation) |
| Multiple directions | More than one filtered direction detected | Invalid |
| Single direction | Consistent motion in one direction | Clockwise or anticlockwise |

The classifier's thresholds were tuned via grid search on 876 labeled video examples, achieving **98.3% accuracy**. Only `"clockwise"` is accepted as valid — anticlockwise and all invalid classifications trigger a retry.

> **Key Design Decision:** The optical flow classifier is a traditional CV approach, not an ML model or LLM call. This is deliberate. Rotation direction is an objective, measurable property. A deterministic algorithm is faster, cheaper, and more reliable than a vision-language model for this specific task.

## Validation: Level 2 — Glitch Detection

Videos that pass the rotation check proceed to **semantic glitch detection** using Gemini 3 Flash as a vision evaluator.

### What It Detects

The glitch detector is calibrated to catch problems that break the illusion of a real product rotating in a studio:

| Category | Examples | Severity |
|----------|----------|----------|
| **Direction reversals** | Product visibly changes rotation direction mid-spin | INVALID |
| **Unnatural transformations** | Text/logos appearing mirrored; screens showing on both sides of a device; features appearing or disappearing | INVALID |
| **Breaking the spin** | Sudden zoom changes; product jumping to frame edge; tilting or teleportation | INVALID |
| **Discontinuities** | Sudden cuts or jumps in the rotation flow | INVALID |

### What It Accepts

Equally important is what the detector is calibrated to *accept*:

| Category | Examples | Severity |
|----------|----------|----------|
| Minor surface imperfections | Slight texture inconsistencies | VALID |
| Lighting variations | Gradual brightness changes during rotation | VALID |
| Reflections and shadows | Natural-looking light interactions | VALID |
| Minor wobbles | Small hesitations in otherwise smooth rotation | VALID |

### Configuration

| Parameter | Value | Rationale |
|-----------|-------|-----------|
| Frame sampling | 2fps (from 24fps source) | Sufficient temporal resolution at lower cost |
| Temperature | 0 | Deterministic evaluation |
| Max output tokens | 300 | Short, structured response |
| Response format | JSON (`{explanation, is_valid}`) | Machine-parseable output |

The two-level validation approach — cheap deterministic check followed by expensive semantic check — is a pattern that recurs throughout the architecture. It optimizes for cost while maintaining quality coverage.

## End-to-End Flow Summary

| Step | Technology | Cost | Latency |
|------|-----------|------|---------|
| Background removal | Vertex AI Segmentation (+ rembg fallback) | Low | ~2s per image |
| Upscaling | Imagen 4.0 | Medium | ~5s per image |
| Description generation | Gemini 2.5 Flash | Low | ~1s |
| Video generation | Veo 3.1 | High | ~60–90s |
| Rotation check | Optical flow (local) | Free | <1s |
| Glitch detection | Gemini 3 Flash | Low | ~5s |

**Typical pipeline time:** 90–120 seconds for a single successful generation. With retries, up to ~5 minutes worst case.

**Success rate:** With input optimization and two-level validation, the pipeline produces acceptable spinning videos for the vast majority of generic products without manual intervention.
